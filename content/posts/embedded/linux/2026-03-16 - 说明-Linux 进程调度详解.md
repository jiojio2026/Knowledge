# Linux 进程调度详解

## 一、核心概念

**进程调度的本质**：决定哪个进程在何时占用 CPU，以及占用多久。

Linux 是多任务系统，但 CPU 核心数是有限的。调度器的任务就是在多个可运行进程之间公平、高效地分配 CPU 时间。

---

## 二、调度器演进史

| 版本 | 调度器 | 核心特点 |
|------|--------|----------|
| 2.4 之前 | 简单轮转 | O(n) 复杂度，进程多了就慢 |
| 2.6.0-2.6.22 | O(1) Scheduler | 每个 CPU 一个运行队列，优先级分层 |
| 2.6.23+ | CFS (Completely Fair Scheduler) | 红黑树维护虚拟运行时间，追求绝对公平 |
| 4.0+ | CFS + 实时调度 | 混合调度，实时进程优先 |

**现在用的都是 CFS**，除非你搞实时系统。

---

## 三、CFS 的核心思想

### 3.1 虚拟运行时间 (vruntime)

CFS 不记录进程"实际跑了多久"，而是记录**虚拟运行时间**：

```
vruntime = 实际运行时间 × (1024 / 进程权重)
```

- 优先级高的进程，权重高，vruntime 增长慢
- 优先级低的进程，权重低，vruntime 增长快

**调度决策**：总是选择 vruntime 最小的进程运行。

### 3.2 红黑树维护

所有可运行进程按 vruntime 排序，存放在红黑树中：

```
        进程 B (vruntime=100)
       /                      \
进程 A (50)              进程 C (200)
```

- 最左节点 = vruntime 最小 = 下一个被调度的进程
- 插入、删除、查找都是 O(log n)

### 3.3 时间片概念的变化

CFS **没有固定的时间片**。每个进程能跑多久取决于：

```
目标延迟 (target_latency) / 可运行进程数
```

例如：
- 目标延迟 = 20ms
- 10 个进程 → 每个跑 2ms
- 100 个进程 → 每个跑 0.2ms (但有最小粒度限制，默认 0.75ms)

## 四、进程优先级

### 4.1 Nice 值 (普通进程)

| Nice 值 | 权重 | 含义 |
|---------|------|------|
| -20 | 88761 | 最高优先级 |
| 0 | 1024 | 默认 |
| 19 | 15 | 最低优先级 |

```bash
# 查看进程 nice 值
ps -eo pid,nice,comm

# 启动时设置 nice
nice -n 10 ./myapp

# 运行时调整
renice -n 5 -p 1234
```

### 4.2 实时优先级 (SCHED_FIFO / SCHED_RR)

| 优先级范围 | 含义 |
|------------|------|
| 1-99 | 实时进程，数字越大优先级越高 |

实时进程**永远优先于**普通进程 (CFS)。

---

## 五、调度策略

| 策略 | 宏定义 | 适用场景 |
|------|--------|----------|
| SCHED_NORMAL | 普通进程 | 绝大多数应用 |
| SCHED_FIFO | 实时，先到先服务 | 硬实时任务，不主动让出就是一直跑 |
| SCHED_RR | 实时，轮转 | 实时任务，有时间片限制 |
| SCHED_BATCH | 批处理 | 后台计算，不追求响应速度 |
| SCHED_IDLE | 极低优先级 | 后台清理、日志压缩等 |
| SCHED_DEADLINE | 截止期限调度 | 有严格时间约束的任务 |

## 六、关键数据结构

### 6.1 task_struct

每个进程都有一个 `task_struct`，包含调度相关信息：

```c
struct task_struct {
    pid_t pid;
    int prio;              // 动态优先级
    int static_prio;       // 静态优先级 (nice 决定)
    int normal_prio;       // 常规优先级
    u64 vruntime;          // CFS 虚拟运行时间
    struct sched_entity se; // 调度实体
    // ...
};
```

### 6.2 运行队列 (rq)

每个 CPU 核心有自己的运行队列：

```c
struct rq {
    struct cfs_rq cfs;     // CFS 运行队列
    struct rt_rq rt;       // 实时运行队列
    struct task_struct *curr; // 当前运行进程
    // ...
};
```

### 6.3 CFS 队列 (cfs_rq)

```c
struct cfs_rq {
    u64 min_vruntime;      // 队列中最小的 vruntime
    struct rb_root tasks_timeline; // 红黑树根
    unsigned int nr_running; // 队列中进程数
    // ...
};
```

## 七、调度过程

### 7.1 调度触发时机

1. **主动调度**：进程调用 `schedule()`，比如等待 I/O、锁、sleep
2. **被动调度**：时间片用完、被抢占、优先级变化
3. **中断返回**：从中断处理程序返回时检查是否需要调度

### 7.2 调度流程

```
1. 保存当前进程上下文
2. 调用 pick_next_task() 选择下一个进程
   - 先检查实时队列 (rt_rq)
   - 再检查 CFS 队列 (cfs_rq)
3. 调用 context_switch() 切换上下文
4. 恢复新进程上下文，开始执行
```

### 7.3 上下文切换开销

```bash
# 查看上下文切换次数
vmstat 1

# 查看每个进程的自愿/非自愿切换
pidstat -w 1
```

**优化建议**：
- 减少锁竞争
- 避免频繁 sleep/wakeup
- CPU 亲和性绑定 (taskset)

## 八、多核调度

### 8.1 负载均衡

Linux 会定期在 CPU 之间迁移进程，保持负载均衡：

- **周期**：每 200ms 检查一次
- **策略**：从忙的 CPU 迁移到 idle 的 CPU
- **代价**：缓存失效，所以不会频繁迁移

### 8.2 CPU 亲和性

```bash
# 查看进程绑定的 CPU
taskset -cp 1234

# 绑定进程到特定 CPU
taskset -c 0,1 ./myapp

# 内核中设置
sched_setaffinity(pid, mask);
```

**适用场景**：
- 高性能计算 (减少缓存失效)
- 实时系统 (隔离关键任务)
- NUMA 架构 (本地内存访问)

## 九、调试与分析工具

### 9.1 /proc 接口

```bash
# 查看进程调度统计
cat /proc/1234/sched

# 查看系统调度统计
cat /proc/schedstat
```

### 9.2 perf 工具

```bash
# 记录调度事件
perf record -e sched:sched_switch -a

# 分析调度延迟
perf sched latency

# 查看 CPU 迁移
perf record -e sched:sched_migrate_task -a
```

### 9.3 ftrace

```bash
# 启用调度跟踪
echo 1 > /sys/kernel/debug/tracing/events/sched/enable

# 查看调度轨迹
cat /sys/kernel/debug/tracing/trace_pipe
```

## 十、常见问题与优化

### 10.1 优先级反转

**问题**：低优先级进程持有锁，阻塞高优先级进程。

**解决**：
- 优先级继承 (PI) 互斥锁
- 使用 `pthread_mutexattr_setprotocol(PTHREAD_PRIO_INHERIT)`

### 10.2 延迟抖动

**现象**：响应时间不稳定。

**排查**：
```bash
# 查看调度延迟
perf sched record
perf sched latency --sort avg
```

**优化**：
- 调整 `sched_min_granularity_ns`
- 使用 `SCHED_DEADLINE` 策略
- 隔离 CPU 核心 (`isolcpus` 内核参数)

### 10.3 实时性保证

```bash
# 设置实时调度策略
chrt -f 99 ./realtime_app

# 锁定内存，避免缺页中断
mlockall(MCL_CURRENT | MCL_FUTURE)
```

## 十一、内核参数调优

```bash
# 查看当前调度参数
cat /proc/sys/kernel/sched_min_granularity_ns
cat /proc/sys/kernel/sched_latency_ns
cat /proc/sys/kernel/sched_wakeup_granularity_ns

# 临时调整
sysctl -w kernel.sched_min_granularity_ns=1000000
```

| 参数 | 默认值 | 含义 |
|------|--------|------|
| sched_min_granularity_ns | 750000 (0.75ms) | 最小运行时间 |
| sched_latency_ns | 24000000 (24ms) | 目标延迟 |
| sched_wakeup_granularity_ns | 1000000 (1ms) | 唤醒粒度 |

---

## 总结

1. **CFS 是核心**：用 vruntime 实现公平调度
2. **实时优先**：SCHED_FIFO/RR 永远优先于 CFS
3. **多核负载均衡**：自动迁移，但有缓存代价
4. **调试工具丰富**：perf、ftrace、/proc
5. **调优需谨慎**：默认值适合大多数场景，改了要有依据
